# TDD / テスト駆動開発の進め方

## TDDにおけるタスクの進め方

- RED->GREE->REFACTORの順でタスクを進める
  - RED：次の目標を考え，その目標を示すテストを書く．書いたテストが失敗することを確認する．
  - GREEN：目的のコードを書き，書いたテストが成功するようになることを確認する．
  - REFACTOR：テストが通る状態を維持しつつリファクタリングを行う．
  - この流れでタスクを進めることを，**必ず**TODOに明記してタスクを推進すること．

## 注意点

### REDフェーズの考え方

#### REDフェーズでテストを書く目的

**失敗するテストを書くこと自体を目的化しないこと．**
このフェーズは「目的のコードを書いたら通過するようになるテストを書くこと」に意味がある．
**ただ失敗するテストを書けば良いのではない．**
以下のような実装は全くの無意味である．

```
# 失敗はするがTDDのとして何ら意味を成さないテスト
expect(false).toBe(true);
```

#### 目的のコード実装前でもテストを書く

前述の通り，このフェーズは「目的のコードを書いたら通過するようになるテストを書くこと」に意味がある．
従って，**○○は未実装なのでテスト実装はしない**という考え方は言語道断である．

### コメントの考え方

**REFACTORフェーズ完了まで進んだときにも意味のあるコメントを書くこと**
つまり，「これは未実装なので今はテストが失敗する」などというコメントは全くの無意味である．
